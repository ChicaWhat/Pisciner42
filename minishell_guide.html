Guía técnica detallada de la arquitectura completa: desde que el usuario pulsa Enter hasta que el proceso termina. Con especial énfasis en el executor, pipelines, file descriptors y señales.

01

Visión General de la Arquitectura

Minishell tiene 5 grandes fases que ocurren en secuencia cada vez que el usuario introduce un comando. Cada fase transforma los datos de una forma específica:

INPUT Usuario escribe: ls -la | grep foo > out.txt
PRE-PARSER Validación de sintaxis: quotes, pipes, redirects
TOKENIZER String → Lista enlazada de t_token
EXPANDER Expande $VAR y $? en los tokens
PARSER Tokens → Lista enlazada de t_cmd
EXECUTOR Ejecuta: fork, pipe, exec, wait
💡

Las estructuras clave son t_mini (estado global del shell), t_token (lista de tokens) y t_cmd (lista de comandos ya parseados con sus FDs y args listos).

02

Flujo Principal — minishell.c

El bucle principal está en shell_loop() que llama repetidamente a minishell_iteration():

// Cada iteración del shell hace esto: minishell_iteration() ├─ set_signals_interactive() // SIGINT → reprint prompt, SIGQUIT → ignorar ├─ readline("Minishell$> ") // Lee input del usuario (bloquea) ├─ g_signal = 0 // Reset señal global ├─ input_to_token(input, ...) // Pre-parse + tokenizar → mini->tokens ├─ process_and_execute(mini) // Expandir + parsear + ejecutar └─ free_iteration_data(mini) // Limpiar tokens y cmds
⚠️

readline() devuelve NULL cuando el usuario pulsa Ctrl+D (EOF). En ese caso el shell imprime "exit" y termina limpiamente.

process_and_execute() es el corazón que encadena las fases finales:

process_and_execute(mini) 1. ft_lstiter(tokens, expand_checker) // Marca qué tokens tienen $VAR 2. ft_lstiter(tokens, heredoc_bf_dollar) // Heredoc delimiters NO expanden 3. expander(mini) // Sustituye $VAR por su valor 4. ft_lstiter(tokens, trim_quotes) // Elimina comillas del contenido 5. init_cmd(&mini) // Tokens → t_cmd (parser) 6. find_full_path(mini) // Busca /usr/bin/ls etc. 7. executor_dispatch(mini) // ¡A ejecutar!
03

Tokenizer — De String a Tokens

El tokenizer transforma el string crudo en una lista enlazada de t_token. Hay 3 funciones principales:

smart_split()

Divide el input en un array de strings, respetando comillas y operadores (|, <, >, <<, >>).

add_token_back()

Por cada string del array, crea un nodo t_token con su tipo y lo añade al final de la lista.

Tipos de token:

TipoValorStringDescripción
WORD0ls, -la, fooPalabra o argumento normal
PIPE1|Separador de comandos en pipeline
REDIR_IN2<Redirección de entrada
REDIR_OUT3>Redirección de salida (truncate)
HEREDOC4<<Heredoc
APPEND5>>Redirección de salida (append)
ENV_VAR6$HOMEVariable de entorno (tras expansión)
EXIT_STATUS7$?Exit status del último comando
💡

smart_split usa word_count() primero para saber cuánta memoria alocar, y luego fill_mem() para rellenar el array. Los operadores (|, <, >) siempre se separan como tokens propios aunque no tengan espacios.

Ejemplo: echo $HOME | grep /usr

// Lista resultante de tokens: [WORD:"echo"] → [ENV_VAR:"$HOME"] → [PIPE:"|"] → [WORD:"grep"] → [WORD:"/usr"] → NULL
04

Parser — De Tokens a Comandos

El parser (init_cmd()) recorre la lista de tokens y la convierte en una lista enlazada de t_cmd. Cada t_cmd representa un comando separado por |.

// Estructura t_cmd: typedef struct s_cmd { char **args; // Array de argumentos: ["ls", "-la", NULL] char *cmd_path; // Path completo: "/bin/ls" char *infile; // Fichero de entrada (< file) char *outfile; // Fichero de salida (> file) int fd_in; // FD de entrada (0=stdin, o FD heredoc) int fd_out; // FD de salida (1=stdout, o FD archivo) int append; // 1 si >> (append), 0 si > pid_t pid; // PID del proceso hijo struct s_cmd *next; // Siguiente comando en pipeline } t_cmd;

La función handler_redirects() procesa cada token del comando actual:

TOKEN ¿Tipo del token actual?
REDIR tipo 2–5 → set_redirects(): abre FDs / heredoc
WORD tipo 0,6,7 → cmd->args[i++] = strdup(content)
PIPE tipo 1 → fin del cmd actual, crear nuevo t_cmd
⚠️

Importante: cuando hay un << (heredoc), el parser llama a handle_heredoc() en ese mismo momento (durante el parsing, antes de ejecutar). Crea un archivo temporal, lee líneas hasta el delimitador, y guarda el FD del archivo en cmd->fd_in.

05

Expander — Variables de Entorno

El expander sustituye $VARIABLE y $? por sus valores reales. Se ejecuta antes del parser, directamente sobre los tokens.

expander(mini) └─ Para cada token con expand==1: 1. get_after_dollar(content) // Encuentra posición del $ 2. get_var_name(str) // Extrae nombre: "HOME", "?" 3. get_env_content(name, mini) // Busca en lista t_env 4. replace_string(...) // Construye nuevo string 5. Repite hasta no haber más $ en el token 6. Si content == "" → eliminar token de la lista

¿Cuándo NO expande?

Si el $ está dentro de comillas simples (''), la función expand_checker() no pone expand=1. Las comillas dobles sí permiten expansión.

Heredoc sin expansión

heredoc_bf_dollar() detecta si un token es el delimitador de un heredoc (<<) y fuerza expand=0 en él, comportamiento correcto de bash.

06

Executor — El Corazón del Shell

El executor recibe la lista de t_cmd ya procesada y la ejecuta. El punto de entrada es executor_dispatch():

executor_dispatch(mini) ├─ if (cmd->next == NULL) │ └─ handle_single_cmd() // Un solo comando └─ else └─ handle_pipeline_loop() // Varios comandos con pipes

Caso 1: Comando Único — handle_single_cmd()

Un comando único puede ser un builtin padre (cd, export, unset, exit) o un proceso externo.

DISPATCH handle_single_cmd(mini, cmd)
CHECK ¿Es builtin Y es parent_builtin? (cd/export/unset/exit)
SÍ → PARENT execute_builtin_in_parent() — sin fork, proceso padre
NO → fork() Crear proceso hijo → single_cmd_child_process()
PADRE single_cmd_parent_wait() → waitpid()
🚨

¿Por qué cd, export, unset y exit NO se forkean? Porque necesitan modificar el estado del proceso padre. Si haces cd /tmp en un hijo, el padre no cambia de directorio. Lo mismo con las variables de entorno y el exit.

🔵 Proceso PADRE — builtins en padre
  • backup_std_fds() — guarda stdin/stdout
  • setup_redirections(cmd) — aplica FDs
  • execute_builtin_func() — ejecuta builtin
  • restore_std_fds() — restaura stdin/stdout
  • mini->exit_status = resultado
🟢 Proceso HIJO — fork()
  • set_signals_default() — restaura señales
  • setup_redirections(cmd) — aplica FDs
  • ¿builtin? → execute_builtin_func() + exit
  • ¿externo? → execute_system_binary()
  • execve(path, args, env) — reemplaza proceso
  • Si falla → exit(126 o 127)

La llamada execve() — El Momento Clave

// execute_system_binary() — en el proceso HIJO env = env_to_array(mini->env) // t_env* → char** para execve execve(cmd->cmd_path, cmd->args, env) // Si llegamos aquí, execve FALLÓ perror(cmd->args[0]) exit(126) // Permission denied / not executable // Nota: si cmd_path es NULL (comando no encontrado): ft_fprintf(2, ERR_CMD_NOT_FOUND, cmd->args[0]) exit(127) // Command not found
💡

execve reemplaza el proceso completamente. El código del hijo después de execve nunca se ejecuta (salvo si falla). La memoria, código y todo del hijo es sustituido por el programa nuevo.

07

Pipelines — handle_pipeline_loop()

Este es el mecanismo más complejo. Cuando hay N comandos separados por |, el shell necesita N-1 pipes y N procesos hijos.

La función handle_pipeline_loop() itera sobre la lista de comandos:

handle_pipeline_loop(mini) prev_pipe = -1 // Al principio no hay pipe anterior current_cmd = mini->cmds while (current_cmd != NULL): 1. init_pipe() // Si hay cmd siguiente: pipe(pipe_fd) 2. spawn_process() // fork() → hijo o padre if (hijo / pid == 0): 3a. pipe_child_process() // Conectar FDs + ejecutar else (padre / pid > 0): 3b. pipe_parent_process() // Cerrar extremos usados current_cmd = current_cmd->next 4. wait_all_children() // Esperar a TODOS los hijos

Regla crítica de pipes: Si el proceso padre no cierra pipe[1] (el extremo de escritura) después de hacer fork, el proceso hijo lector nunca recibirá EOF y quedará bloqueado esperando datos. Siempre hay que cerrar todos los extremos que no se usen.

08

File Descriptors y Procesos Hijo/Padre

¿Qué es un File Descriptor (FD)?

Un FD es un número entero que el kernel asigna a cada archivo/pipe/socket abierto. Cada proceso tiene su propia tabla de FDs:

0 → STDIN_FILENO — entrada estándar por defecto: teclado
1 → STDOUT_FILENO — salida estándar por defecto: terminal
2 → STDERR_FILENO — errores por defecto: terminal
3+ → FDs de archivos, pipes, heredocs... abiertos por el programa

dup2() — La Función Mágica

dup2(oldfd, newfd) hace que newfd apunte al mismo archivo que oldfd. Si newfd ya estaba abierto, lo cierra primero. Así se redirigen stdin/stdout:

// Redirigir stdout al extremo de escritura de un pipe: dup2(pipe_fd[1], STDOUT_FILENO) // FD 1 ahora → pipe[1] close(pipe_fd[1]) // Cerramos el duplicado extra // Ahora cualquier printf() o write(1,...) va al pipe // Redirigir stdin al extremo de lectura del pipe anterior: dup2(prev_pipe, STDIN_FILENO) // FD 0 ahora → pipe anterior[0] close(prev_pipe) // Cerramos el duplicado extra // Ahora cualquier read(0,...) lee del pipe

Ciclo de vida de FDs en pipe_child_process()

pipe_child_process(mini, cmd, prev_pipe, pipe_fd) // 1. Restaurar señales a default (el hijo no ignora nada) set_signals_default() // 2. Conectar stdin al pipe anterior (si existe) if (prev_pipe != -1): dup2(prev_pipe, STDIN_FILENO) close(prev_pipe) // 3. Conectar stdout al pipe siguiente (si no es el último) if (cmd->next): close(pipe_fd[0]) // El hijo no lee de este pipe dup2(pipe_fd[1], STDOUT_FILENO) close(pipe_fd[1]) // 4. Ejecutar el comando (aplica redirections, luego exec) execute_command_node(mini, cmd) // ↑ Esta función llama execve → el hijo es reemplazado

Ciclo de vida de FDs en pipe_parent_process()

pipe_parent_process(cmd, &prev_pipe, pipe_fd) // Cerrar el extremo del pipe ANTERIOR que ya no necesitamos if (*prev_pipe != -1): close(*prev_pipe) // El hijo ya lo tiene, nosotros no lo usamos // Si hay siguiente comando: guardar el extremo de lectura if (cmd->next): close(pipe_fd[1]) // El hijo escribe aquí, nosotros cerramos *prev_pipe = pipe_fd[0] // Guardar lectura para el siguiente hijo
✅

La variable prev_pipe es el hilo conductor: empieza en -1 y en cada iteración se actualiza con el pipe_fd[0] del pipe recién creado, para que el siguiente hijo lo use como stdin.

¿Qué pasa con los FDs de redirecciones?

La función setup_redirections() se ejecuta dentro de cada hijo:

Entrada (fd_in o infile)

Si fd_in > 0 (heredoc): dup2(fd_in, 0).

Si infile existe: abre el archivo con O_RDONLY y hace dup2.

La redirección de pipe ya se hizo antes en pipe_child_process.

Salida (fd_out o outfile)

Si fd_out > 1: dup2(fd_out, 1).

Si outfile existe: abre con O_WRONLY|O_CREAT|O_TRUNC (o O_APPEND) y hace dup2.

Prioridad de redirections: Las redirections explícitas (> file) sobreescriben las de la pipe. Si tienes cmd1 | cmd2 > file, cmd2 recibe datos de la pipe en stdin, pero escribe en el archivo en vez de en la siguiente pipe.

wait_all_children() — Recoger el Exit Status

wait_all_children(mini) set_signals_ignore() // Padre ignora señales mientras espera for each cmd in mini->cmds: waitpid(cmd->pid, &status, 0) // Bloqueante: espera a que termine // Analizar el status del ÚLTIMO proceso if WIFEXITED(status): exit_status = WEXITSTATUS(status) // Terminó normalmente: exit(N) else if WIFSIGNALED(status): g_signal = 128 + WTERMSIG(status) // Murió por señal: 128+N if SIGINT: write("\n") if SIGQUIT: write("Quit (core dumped)")
💡

En bash, el exit status de un pipeline es el del último comando. Por eso solo se guarda el status del último waitpid. Los anteriores se esperan pero su status se descarta.

09

Señales — Manejo de Ctrl+C y Ctrl+\

Las señales son notificaciones asíncronas enviadas a procesos. En minishell hay 5 estados distintos con diferentes configuraciones:

INTERACTIVE

Ctrl+C → handler_sigint_interactive: nueva línea + reprint prompt

Ctrl+\ → SIG_IGN (ignorado)

Estado: esperando input del usuario

CHILD

Ctrl+C → handler_signals_child: escribe "\n"

Ctrl+\ → SIG_IGN

Estado: padre esperando a hijo único

DEFAULT

Ctrl+C → SIG_DFL (comportamiento default)

Ctrl+\ → SIG_DFL

Estado: dentro del proceso hijo (antes de exec)

HEREDOC

Ctrl+C → handler_sigint_heredoc: cierra stdin

Ctrl+\ → SIG_IGN

Estado: leyendo heredoc

IGNORE

Ctrl+C → SIG_IGN

Ctrl+\ → SIG_IGN

Estado: padre esperando a todos los hijos del pipeline

La Variable Global g_signal

int g_signal; // Variable global — única permitida en minishell // Se setea en los handlers: handler_sigint_interactive(int signum): g_signal = signum // signum = SIGINT = 2 write(1, "\n", 1) rl_on_new_line() rl_replace_line("", 0) rl_redisplay() // Redibuja el prompt vacío // Se usa para detectar interrupción: if (g_signal != 0): mini->exit_status = 130 // 128 + 2 (SIGINT)
⚠️

Los signal handlers son funciones muy restringidas — solo pueden llamar a funciones async-signal-safe. Por eso solo hacen write() y modifican la variable global. El procesamiento real ocurre en el hilo principal al revisar g_signal.

Flujo de Señales en Heredoc

USER Pulsa Ctrl+C durante heredoc
HANDLER handler_sigint_heredoc: g_signal=2, cierra STDIN
READ readline() devuelve NULL (stdin cerrado)
CHECK hd_read_loop: g_signal != 0 → hd_signal_cleanup()
CLEANUP Cierra FD, unlink temp, restaura stdin desde backup, return -1
RESTORE Vuelve al prompt interactivo normal

Por eso existe mini->stdin_backup = dup(STDIN_FILENO) antes del heredoc — para poder restaurar stdin después de que el handler lo cierre.

Builtins — Comandos Internos

Los builtins son comandos implementados directamente en el shell. No son procesos externos.

Builtin¿En padre?¿Por qué?Exit codes
cd Sí Cambia el directorio del proceso padre (chdir) 0 éxito / 1 error
export Sí Modifica el entorno del proceso padre 0 éxito / 1 inválido
unset Sí Elimina variables del entorno del padre 0 siempre
exit Sí Termina el shell padre N o 0 o 2
echo No Solo escribe en stdout, no afecta estado 0 siempre
pwd No Solo lee CWD, no modifica estado 0 / 1 error
env No Solo imprime, no modifica estado 0 / 127

Detalles de ft_echo

echo [-n] [args...] — La flag -n suprime el salto de línea final. La función check_flag() detecta -n, -nn, -nnn, etc. (solo n's). without_n() imprime sin newline, with_n() imprime con newline.

Detalles de ft_exit

exit [N] — Si N es numérico (is_numeric_av()), sale con código N % 256. Si hay más de un argumento numérico, error "too many arguments". Si el argumento no es numérico, sale con código 2.

Detalles de ft_cd

cd [path] — Sin argumentos: va a $HOME. Con path: chdir(path). Después actualiza $PWD con getcwd y guarda el anterior en $OLDPWD. Usa create_new_env() para actualizar las variables en la lista t_env.

Detalles de ft_export

export [VAR=value] — Sin args: imprime todas las variables ordenadas alfabéticamente (order_env() con bubble sort). Con args: valida que el nombre sea alfanumérico+underscore (is_valid()), luego llama a create_new_env() que actualiza si existe o añade si no.

Diagrama: Builtins en Padre vs Hijo

CALL handle_single_cmd(mini, cmd)
CHECK1 is_builtin(args[0]) && is_parent_builtin(args[0])?
PADRE backup_fds → setup_redirections → execute_builtin_func → restore_fds
HIJO (fork) set_signals_default → setup_redirections → execute_builtin_func → exit(code)
💡

Cuando un builtin se ejecuta en un pipeline (ej: echo hola | cat), siempre se ejecuta en el hijo porque todos los procesos de un pipeline se forkean. Solo cuando es el único comando y es parent_builtin se ejecuta en el padre directamente.

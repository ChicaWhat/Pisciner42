01

Conceptos Fundamentales

🧵 ¿Qué es un Hilo (Thread)?

Un hilo es la unidad mínima de ejecución dentro de un proceso. Varios hilos pueden vivir dentro del mismo proceso y comparten la misma memoria (heap, variables globales, descriptores de archivo), pero cada uno tiene su propia pila (stack) y su propio contador de programa.

Crear un hilo es más barato que crear un proceso (fork), y la comunicación entre hilos es directa porque comparten memoria.

Analogía

Una cocina con varios cocineros. El local (proceso) es el mismo, la nevera y los utensilios (memoria) son compartidos, pero cada cocinero (hilo) trabaja en paralelo en su propia tarea.

🔄 ¿Qué es la Concurrencia?

La concurrencia es la capacidad de que varias tareas progresen al mismo tiempo o aparentándolo. En un sistema con un solo núcleo, los hilos se turnan muy rápido (interleaving). En varios núcleos, pueden correr realmente en paralelo (paralelismo).

En Philosophers, todos los filósofos "viven" al mismo tiempo porque cada uno es un hilo independiente.

Analogía

Varios navegadores abiertos en tu ordenador: aunque solo tengas un procesador, todos "avanzan" porque el SO les va dando turnos de CPU muy rápidamente.

🔒 ¿Qué es un Mutex?

Un mutex (Mutual Exclusion) es un mecanismo de sincronización que garantiza que solo un hilo a la vez pueda acceder a una sección crítica o recurso compartido. Si el mutex está bloqueado, el hilo que intenta bloquearlo de nuevo espera.

En Philosophers, cada tenedor es un mutex: solo un filósofo puede cogerlo a la vez.

Analogía

El baño de una oficina con llave. Solo entra una persona. Las demás esperan fuera. Cuando sale, la siguiente puede entrar y girar el pestillo.

⚠️ ¿Qué es un Data Race?

Un data race ocurre cuando dos hilos acceden a la misma memoria al mismo tiempo y al menos uno de ellos la modifica, sin ninguna sincronización. El resultado es indeterminado y puede cambiar en cada ejecución.

Tu programa no debe tener ningún data race. Protege siempre el acceso a datos compartidos con mutex.

Analogía

Dos personas editando el mismo documento Word sin Google Docs: si las dos guardan a la vez, los cambios de una machacan los de la otra. Resultado impredecible.

💀 Deadlock

Un deadlock es cuando dos o más hilos se bloquean mutuamente esperando recursos que el otro tiene. Ninguno avanza: están atascados para siempre.

Ejemplo en Philosophers: Filósofo 1 coge tenedor izquierdo, Filósofo 2 coge tenedor derecho. Ambos esperan al otro. Nadie come. Todos mueren.

Analogía

Dos coches en una calle estrecha frente a frente. Ninguno puede avanzar porque el otro bloquea el paso. Deadlock vehicular.

⏱️ Timestamp en milisegundos

El proyecto requiere imprimir marcas de tiempo en milisegundos desde el inicio. gettimeofday() devuelve segundos y microsegundos. Necesitas convertir: ms = segundos × 1000 + microsegundos / 1000.

Guarda el tiempo de inicio al arrancar y en cada log resta el tiempo actual menos el de inicio.

Analogía

Un cronómetro de atletismo: no te dice la hora del día, sino cuántos milisegundos han pasado desde que sonó el pistoleto de salida.

⚠ Regla de oro: Todo dato compartido entre hilos que pueda ser modificado DEBE estar protegido por un mutex. Si no lo proteges, tienes un data race y el proyecto falla.
02

Funciones Autorizadas

Memoria memset ▶
void *memset(void *s, int c, size_t n)

Rellena los primeros n bytes de la memoria apuntada por s con el valor c (convertido a unsigned char). Se usa para inicializar bloques de memoria a cero u otro valor.

Ejemplo en Philosophers
t_philo *philo = malloc(sizeof(t_philo)); memset(philo, 0, sizeof(t_philo)); // Inicializa todo a cero // Equivalente a: philo->id = 0; philo->meals = 0; ...
I/O printf ▶
int printf(const char *format, ...)

Imprime texto formateado en stdout. En Philosophers debes imprimir los estados con el formato exacto: timestamp_ms X acción. ⚠ Importante: el printf debe estar protegido con un mutex para que los mensajes no se mezclen entre hilos.

Ejemplo en Philosophers
pthread_mutex_lock(&data->print_mutex); printf("%lld %d is eating\n", get_timestamp(), philo->id); pthread_mutex_unlock(&data->print_mutex);
Memoria malloc ▶
void *malloc(size_t size)

Reserva size bytes en el heap y devuelve un puntero. Devuelve NULL si falla. Toda la memoria reservada debe liberarse con free() antes de que el programa termine. No se permiten memory leaks.

Ejemplo en Philosophers
t_philo *philos = malloc(sizeof(t_philo) * n_philos); if (!philos) return (handle_error("malloc failed"));
Memoria free ▶
void free(void *ptr)

Libera la memoria previamente reservada con malloc. Si ptr es NULL, no hace nada. Debes liberar todo lo que reservas: array de filósofos, array de mutex de tenedores, etc.

Ejemplo en Philosophers
free(data->philos); // Array de filósofos free(data->forks); // Array de mutexes de tenedores data->philos = NULL; // Buena práctica: anular el puntero
I/O write ▶
ssize_t write(int fd, const void *buf, size_t count)

Escribe count bytes del buffer en el descriptor de archivo fd. Más bajo nivel que printf. Útil para mensajes de error: write(2, "Error\n", 6) escribe en stderr. En Philosophers generalmente usarás printf para los logs.

Ejemplo en Philosophers
write(2, "Error: invalid arguments\n", 25); // fd=2 es stderr. fd=1 es stdout.
Tiempo usleep ▶
int usleep(useconds_t usec)

Suspende el hilo durante usec microsegundos (1ms = 1000 µs). ⚠ Problema importante: usleep no es preciso, puede dormir más tiempo del pedido. Para time_to_die muy ajustado, no hagas usleep(time_to_eat * 1000) directamente; usa un bucle que compruebe el tiempo real.

Ejemplo en Philosophers — usleep preciso
void precise_sleep(long ms, t_data *data) { long start = get_time(); while (get_time() - start < ms) { if (is_dead(data)) // alguien murió → parar break; usleep(500); // dormir en trozos de 0.5ms } }
Tiempo gettimeofday ▶
int gettimeofday(struct timeval *tv, struct timezone *tz)

Obtiene el tiempo actual con precisión de microsegundos. La struct timeval contiene tv_sec (segundos) y tv_usec (microsegundos). Pasa NULL al segundo argumento. Es la función base para todo el timing del proyecto.

Ejemplo en Philosophers — función get_time
long get_time(void) { struct timeval tv; gettimeofday(&tv, NULL); return (tv.tv_sec * 1000 + tv.tv_usec / 1000); // Devuelve milisegundos desde epoch UNIX } // Para el timestamp del log: long timestamp = get_time() - data->start_time;
Thread pthread_create ▶
int pthread_create(pthread_t *thread, const pthread_attr_t *attr, void *(*routine)(void *), void *arg)

Crea un nuevo hilo que ejecuta la función routine con el argumento arg. Devuelve 0 en éxito. El identificador del hilo se guarda en thread. Pasa NULL a attr para atributos por defecto. En Philosophers creas un hilo por filósofo.

Ejemplo en Philosophers
int i = 0; while (i < data->n_philos) { if (pthread_create(&philos[i].thread, NULL, philosopher_routine, &philos[i]) != 0) return (handle_error("pthread_create failed")); i++; } // philosopher_routine es la función de vida del filósofo
Thread pthread_detach ▶
int pthread_detach(pthread_t thread)

Marca un hilo como "detached": cuando termina, sus recursos se liberan automáticamente sin necesidad de hacer join. No puedes hacer join a un hilo detached. En Philosophers generalmente usarás pthread_join en vez de detach para esperar a que todos los filósofos terminen.

Cuándo usarlo
// Opción A: detach (no puedes esperar su fin explícitamente) pthread_detach(philo->thread); // Opción B (más común en Philosophers): join pthread_join(philo->thread, NULL); // → espera a que el hilo termine
Thread pthread_join ▶
int pthread_join(pthread_t thread, void **retval)

Espera (bloquea) hasta que el hilo especificado termine. Si retval no es NULL, guarda el valor de retorno del hilo. En Philosophers lo usas al final para esperar a todos los filósofos antes de limpiar y salir. Evita que el main acabe antes que los hilos.

Ejemplo en Philosophers
// Esperar a que todos los filósofos terminen i = 0; while (i < data->n_philos) { pthread_join(philos[i].thread, NULL); i++; } // A partir de aquí podemos liberar memoria
Mutex pthread_mutex_init ▶
int pthread_mutex_init(pthread_mutex_t *mutex, const pthread_mutexattr_t *attr)

Inicializa un mutex. Debe llamarse antes de usar el mutex. Pasa NULL al segundo argumento para atributos por defecto. En Philosophers inicializas un mutex por tenedor (forks) y al menos uno para el print y otro para controlar la muerte.

Ejemplo en Philosophers
// Inicializar mutex de cada tenedor i = 0; while (i < data->n_philos) { if (pthread_mutex_init(&data->forks[i], NULL) != 0) return (handle_error("mutex_init failed")); i++; } pthread_mutex_init(&data->print_mutex, NULL); pthread_mutex_init(&data->death_mutex, NULL);
Mutex pthread_mutex_destroy ▶
int pthread_mutex_destroy(pthread_mutex_t *mutex)

Libera los recursos asociados a un mutex. Debe llamarse cuando ya no se necesita el mutex, normalmente durante la limpieza final. No destruyas un mutex mientras esté bloqueado o haya hilos esperándolo.

Ejemplo en Philosophers
void cleanup(t_data *data) { int i = 0; while (i < data->n_philos) pthread_mutex_destroy(&data->forks[i++]); pthread_mutex_destroy(&data->print_mutex); pthread_mutex_destroy(&data->death_mutex); free(data->forks); free(data->philos); }
Mutex pthread_mutex_lock ▶
int pthread_mutex_lock(pthread_mutex_t *mutex)

Bloquea el mutex. Si ya está bloqueado por otro hilo, espera hasta que se libere. Marca el inicio de la sección crítica. Todo lo que pongas entre lock y unlock solo lo ejecuta un hilo a la vez. ⚠ Nunca hagas lock dos veces seguidas del mismo mutex desde el mismo hilo → deadlock.

Ejemplo en Philosophers — coger un tenedor
// Coger tenedor izquierdo pthread_mutex_lock(&data->forks[philo->left_fork]); print_status(philo, "has taken a fork"); // Coger tenedor derecho pthread_mutex_lock(&data->forks[philo->right_fork]); print_status(philo, "has taken a fork"); // Ahora puede comer (tiene ambos tenedores)
Mutex pthread_mutex_unlock ▶
int pthread_mutex_unlock(pthread_mutex_t *mutex)

Desbloquea el mutex, permitiendo que otro hilo en espera lo bloquee. Marca el fin de la sección crítica. Siempre desbloquea en el mismo hilo que lo bloqueó. Empareja siempre cada lock con su unlock.

Ejemplo en Philosophers — soltar tenedores
// Terminar de comer → soltar tenedores pthread_mutex_unlock(&data->forks[philo->left_fork]); pthread_mutex_unlock(&data->forks[philo->right_fork]); print_status(philo, "is sleeping"); precise_sleep(data->time_to_sleep, data);

03

Roadmap del Proyecto

FASE 1 — FUNDAMENTOS
Paso 1 Parsing

Leer y validar los argumentos de entrada

  • Verificar número de argumentos: deben ser 4 o 5 (con el nombre del programa son 5 o 6 en argv).
    • Si argc es distinto de 5 o 6 → imprimir error y salir
  • Convertir strings a enteros: implementa tu propio ft_atoi o usa atoi, verificando que todos los chars sean dígitos.
    • Rechaza números negativos o cero para n_philos, time_to_die, etc.
    • Rechaza cualquier argumento que no sea puramente numérico
  • Validar rangos: n_philos ≥ 1, todos los tiempos > 0, número de comidas (si existe) > 0.
  • Struct de datos global (t_data): diseña una struct que guarde todos los parámetros.
    • n_philos, time_to_die, time_to_eat, time_to_sleep, must_eat_count
    • start_time, flag de muerte, arrays de filósofos y mutexes
Paso 2 Estructuras

Diseñar las structs del proyecto

  • t_data (datos globales compartidos):
    • Parámetros del programa
    • Puntero al array de filósofos
    • Puntero al array de mutexes de tenedores
    • Mutex para el print (print_mutex)
    • Mutex para el flag de muerte (death_mutex)
    • Flag someone_died (int, protegido por mutex)
    • start_time en ms
  • t_philo (datos de cada filósofo):
    • ID (número del filósofo)
    • Índices de tenedor izquierdo y derecho
    • Timestamp de la última vez que comió
    • Número de comidas realizadas
    • El hilo (pthread_t)
    • Puntero a t_data (para acceder a los datos compartidos)
FASE 2 — INICIALIZACIÓN
Paso 3 Init Mutex

Inicializar todos los mutexes

  • Reservar array de mutexes para tenedores: malloc(sizeof(pthread_mutex_t) * n_philos)
    • Inicializar cada uno con pthread_mutex_init
    • Si alguno falla: destruir los ya creados y liberar
  • Inicializar mutex de print: protegerá el printf para que los logs no se mezclen.
  • Inicializar mutex de muerte: protegerá la lectura/escritura del flag someone_died.
Paso 4 Init Filósofos

Inicializar el array de filósofos

  • Reservar array: malloc(sizeof(t_philo) * n_philos)
  • Asignar valores a cada filósofo:
    • id = i + 1 (del 1 al N)
    • left_fork = i, right_fork = (i + 1) % n_philos
    • last_meal_time = start_time (acaba de "comer" al nacer)
    • meals_eaten = 0
    • puntero a data
  • Caso especial: 1 solo filósofo. Solo tiene un tenedor → nunca puede comer → muere. Maneja este caso especialmente en su rutina.
FASE 3 — EL CORAZÓN DEL PROYECTO
Paso 5 Rutina Filósofo

La función que ejecuta cada hilo

  • Estructura del bucle principal: mientras nadie haya muerto, repetir: comer → dormir → pensar.
    • Antes de cada acción: comprobar si alguien murió (is_simulation_over())
    • Si la simulación acabó → salir de la rutina
  • Fase COMER:
    • Lock tenedor izquierdo → print "taken a fork"
    • Lock tenedor derecho → print "taken a fork"
    • Actualizar last_meal_time (¡con mutex!)
    • Print "is eating"
    • precise_sleep(time_to_eat)
    • Incrementar meals_eaten
    • Unlock tenedor derecho, unlock tenedor izquierdo
  • Fase DORMIR: print "is sleeping" → precise_sleep(time_to_sleep)
  • Fase PENSAR: print "is thinking" (sin sleep fijo, o sleep calculado para evitar starvation)
  • Anti-deadlock: filósofos pares empiezan con un pequeño usleep(1) para desincronizarse, o alternan el orden de coge-tenedores.
Paso 6 Monitor

Hilo o bucle vigilante de la muerte

  • El monitor comprueba periódicamente (cada ~1ms) si algún filósofo lleva más de time_to_die ms sin comer.
    • Iterar por todos los filósofos
    • Leer last_meal_time con mutex
    • Si now - last_meal_time > time_to_die → ese filósofo muere
  • Cuando detecta muerte:
    • Bloquear death_mutex y poner someone_died = 1
    • Imprimir "X died" (con print_mutex)
    • Terminar el monitor
  • Comprobar condición de victoria: si must_eat_count está definido y todos los filósofos alcanzaron ese número → poner someone_died = 1 (stop sin muerte).
  • Opciones de implementación:
    • Opción A: hilo separado solo para monitorizar
    • Opción B: el main monitoriza en un bucle tras crear todos los hilos
Paso 7 Lanzar Hilos

Crear todos los hilos y arrancar la simulación

  • Guardar start_time antes de crear ningún hilo.
  • Loop pthread_create para cada filósofo.
  • Desincronización: para evitar deadlock, los filósofos con id par pueden hacer usleep(1000) al inicio de su rutina.
  • Lanzar el monitor (hilo separado o bucle en main).
  • Loop pthread_join para esperar a todos.
FASE 4 — CIERRE Y CALIDAD
Paso 8 Limpieza

Liberar todos los recursos

  • Destruir todos los mutexes (tenedores, print, death) con pthread_mutex_destroy.
  • Liberar toda la memoria reservada con malloc: array de filósofos, array de forks.
  • Verificar con valgrind que no hay leaks: valgrind --leak-check=full ./philo ...
  • Verificar data races con: valgrind --tool=helgrind ./philo ...
Paso 9 Makefile

Crear el Makefile correcto

  • Reglas obligatorias: $(NAME), all, clean, fclean, re
  • Flags de compilación: -Wall -Werror -Wextra
  • Linker flag para pthreads: añadir -lpthread al compilar
  • No debe hacer relink si los archivos no han cambiado
Paso 10 Testing

Casos de prueba esenciales

  • ./philo 1 800 200 200 → el filósofo debe morir (solo tiene un tenedor)
  • ./philo 5 800 200 200 → nadie debe morir nunca
  • ./philo 5 800 200 200 7 → para cuando cada uno come 7 veces
  • ./philo 4 410 200 200 → nadie debe morir
  • ./philo 4 310 200 100 → un filósofo debe morir
  • ./philo 2 800 200 200 → caso con solo 2 filósofos, comen alternando
BONUS — SOLO SI LA PARTE OBLIGATORIA ES PERFECTA
Bonus Semáforos

philo_bonus: procesos + semáforos en vez de hilos + mutex

  • Cada filósofo es un proceso creado con fork(), no un hilo.
  • Los tenedores se representan con un semáforo (sem_open) de valor N_philos.
  • El proceso main no es filósofo: solo crea hijos y espera.
  • Nuevas funciones: fork, kill, exit, waitpid, sem_open, sem_close, sem_post, sem_wait, sem_unlink.
